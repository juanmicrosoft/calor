# RFC: Elide `§/C` closer for single-argument `§C` call expressions

**Status:** Draft (v2 — rewritten after architectural-fiction audit; see §0.1)
**Owner:** Calor language team
**Target release:** v0.6.x
**Companion RFCs:** `v0.6-bind-inference-formalization.md`

## 0. Front matter

### 0.1 Correction note (v2)

The v1 draft of this RFC ("Token Economics v1 — Elide `§A` / `§/C` for single-argument calls")
overclaimed novelty and quoted fabricated frequency counts. A code-and-corpus audit
showed that **both** of the proposed elisions are already implemented in the parser
for statement context, and that the corpus counts in v1 were ~450× the true values.

What changed in v2:

- **Scope narrowed** to one concrete new behavior: dropping the `§/C` closer on
  single-argument call **expressions** (not statements — already supported).
- **All frequency counts replaced** with measured values and reproducible commands
  (see §1.3).
- **Token-economics estimate downgraded** from the v1 "+5 % `TokenEconomics`" to
  a measured corpus saving of **up to 951 tokens out of 162 961 corpus tokens
  (~0.58 %)** for the conservative proposal (drop `§/C` only), or **up to
  1 473 tokens (~0.90 %)** if the emitter is also updated to drop `§A` on
  one-arg calls (a separate, already-parser-supported change). Both numbers
  use the same tokenization as the September benchmark (see §1.3 for the
  tokenizer logic).
- **Parser-precedence problem** is now the central design question (§3.2), not
  a footnote labelled "mitigation". The v1 draft should not be used as design
  or measurement evidence.

### 0.2 Scope (one- and zero-argument calls)

The grammar change covers both **zero-argument** (`§C{Vec.empty}` with no
`§/C`) and **one-argument inline** (`§C{Console.FormatHello} STR:"World"`
with no `§/C`) calls. The multi-arg form is unchanged.

The corpus-measured win is modest but not negligible:

- **~0.58 %** under the September benchmark tokenizer for the conservative
  proposal (drop `§/C` only).
- **~0.90 %** if combined with an emitter change to drop `§A` on one-arg
  calls — a transformation the parser already supports for statement
  context (Parser.cs:1374-1382) but does not yet support for expression
  context.

Beyond the corpus, each elided `§/C` and `§A` removes 3 + 2 = 5 tokens at
LLM call sites for AI-generated Calor, so the runtime cost story is more
favourable than corpus measurement alone suggests.

## 1. Existing behavior (verified against the parser source)

### 1.1 Statement-context call (already supports implicit close)

```calr
§F{f001:greet:str:pub}
  §C{Console.WriteLine} "Hello"      # works today, no §/C needed
§/F
```

Source: `src/Calor.Compiler/Parsing/Parser.cs:1360`–`1406` (`ParseCallStatement`).
Lines 1374–1382 explicitly handle the implicit-close form:

```csharp
// Support implicit closing - if a single expression follows without §A, treat it as the argument
// §C[Console.WriteLine] "Hello" is equivalent to §C[Console.WriteLine] §A "Hello" §/C
if (IsExpressionStart() && !Check(TokenKind.Arg))
{
    arguments.Add(ParseExpression());
    var span = startToken.Span.Union(arguments[0].Span);
    return new CallStatementNode(span, target, fallible, arguments, attrs);
}
```

### 1.2 Expression-context call (still requires `§/C`)

```calr
§B{greeting} §C{Console.FormatHello} STR:"World" §/C   # §/C required today
```

Source: `Parser.cs:7830`–`7920` (`ParseCallExpression`). Lines 7871–7890 loop
until `§/C` is consumed by `ExpectBlockEnd(TokenKind.EndCall)`. The inner
`else if (IsExpressionStart())` arm at line 7878 already accepts inline
arguments without `§A`, but the trailing `§/C` is unconditional.

### 1.3 Corpus frequency (reproducible)

Counted on the repo at the commit this RFC was prepared against
(`docs/tokenecon-v1-rfcs` branch, rebased on `10583118`):

```powershell
$files = Get-ChildItem -Path samples,tests -Recurse -Filter *.calr -File
$calls   = ($files | ForEach-Object { (Get-Content $_ -Raw) } |
            ForEach-Object { ([regex]::Matches($_, '§C\{')).Count } | Measure-Object -Sum).Sum
$args    = ($files | ForEach-Object { (Get-Content $_ -Raw) } |
            ForEach-Object { ([regex]::Matches($_, '§A(\s|\{)')).Count } | Measure-Object -Sum).Sum
$closers = ($files | ForEach-Object { (Get-Content $_ -Raw) } |
            ForEach-Object { ([regex]::Matches($_, '§/C')).Count } | Measure-Object -Sum).Sum
```

| Quantity | Value |
|---|---:|
| `.calr` files in `samples/` + `tests/` | 1 552 |
| `§C{` openers | **343** |
| `§A` markers | **388** |
| `§/C` closers | **343** |
| Avg `§A` per call | **1.13** |

Call-shape histogram (within `§C{...}…§/C` blocks):

| Shape | Count | Share |
|---|---:|---:|
| Zero-arg (empty body) | 56 | 16.3 % |
| Inline arg without `§A` (`§C{x} expr §/C`) | 0 | 0 % |
| One-arg with explicit `§A` (`§C{x} §A expr §/C`) | 261 | 76.1 % |
| Multi-arg with explicit `§A` (≥ 2 `§A`) | 26 | 7.6 % |
| **Total** | **343** | 100 % |

Two observations from the histogram:

- The "inline arg without `§A`" form is **legal for statement calls** but is
  not emitted by `CalorEmitter` anywhere in the corpus. Of those 343 callers,
  100 % currently use the canonical `§A … §/C` shape.
- 261 of 343 calls (76 %) are exactly one-arg and would benefit from this RFC.

The histogram does **not** distinguish statement-context calls (where
implicit-close is already legal per §1.1) from expression-context calls
(where it is not). Splitting requires walking the bound tree per call site
and was deferred to Phase 0 of the implementation plan (§6).

The "rg" block in v1 (`samples/ 17_412 ; tests/ 132_801`) was a fabrication;
the true count is 343. The v1 estimate of "11 400 calls in the benchmark corpus"
was similarly fictitious.

### 1.4 Benchmark tokenizer

The September benchmark's `TokenEconomics` metric uses the tokenizer at
`src/Calor.Compiler/Evaluation/Metrics/TokenEconomicsCalculator.cs`
(`TokenizeSource`): it splits on whitespace **and** on individual
punctuation / symbol characters. Under that tokenizer:

| String | Tokens |
|---|---:|
| `§C{f}` | 5 (`§`, `C`, `{`, `f`, `}`) |
| `§C{f} §A x §/C` | 11 |
| `§C{f} x` | 6 |
| `§A` | 2 |
| `§/C` | 3 |
| `§B{x:i32}` | 7 |
| `§B{x}` | 5 |

So dropping `§/C` saves **3 tokens** and dropping `§A` saves **2 tokens**.
The benchmark corpus tokenized under the same rules contains **162 961**
tokens across the 1 552 `.calr` files. These two numbers anchor every
estimate in §4.

## 2. Proposal

Allow `§/C` to be elided on a `§C{…}` **call expression** that has exactly one
argument, when that argument is a primary-precedence expression followed by an
end-of-line, `§/F`, `§/L`, `§/I`, `§EL`, `§EI`, `§/M`, `§)` (close-paren), or
any of `§A`, `§C{`, `§/C` (a syntactic boundary that cannot be part of the
argument).

```calr
§B{greeting} §C{Console.FormatHello} STR:"World"      # NEW: no §/C
§B{u}        §C{Map.size}                              # NEW: zero-arg, no §/C
§B{xs}       §C{Vec.empty}                             # NEW: zero-arg, no §/C
§B{ys}       §C{Vec.map} §A xs §A f §/C                # unchanged (multi-arg)
§B{z}        §C{Math.add} (+ a b)                      # Lisp-prefix arg as a single primary expression
```

The statement form is already implicit-closing and is **unchanged**.

### 2.1 Grammar (changes only)

```
call_expression
    : '§C{' member_path '}'                                    # NEW: zero-arg, implicit close
    | '§C{' member_path '}' primary_expr                       # NEW: one inline-arg, implicit close
    | '§C{' member_path '}' '§A' arg                       '§/C'  # unchanged: 1-arg explicit form
    | '§C{' member_path '}' ('§A' arg)+                    '§/C'  # unchanged: multi-arg
    | '§C{' member_path '}' inline_arg_no_prefix           '§/C'  # unchanged: inline + explicit closer
    ;
```

`primary_expr` is at **primary precedence** only — see §3.2.

### 2.2 Emitter

The migration emitter (C# → Calor) emits the elided form only when:

1. The call has 0 or 1 argument.
2. The argument (if any) round-trips at primary precedence — i.e. parsing the
   emitted form yields a `CallExpressionNode` with the same target and same
   single argument.
3. The next token in the output stream is a hard syntactic boundary
   (newline, dedent, or one of the terminators listed in §2).

Gate behind a new emitter option. The current `CalorEmitter` constructor
takes a `ConversionContext?` (see `src/Calor.Compiler/Migration/CalorEmitter.cs:11-22`);
add `UseImplicitCallCloser` to `ConversionContext` rather than introducing a
new `CalorEmitterOptions` class. Default `false` in v0.6.0, flip to `true`
in v0.6.1 after snapshot consumers have updated their golden files.

`CSharpEmitter` is unaffected — it consumes the bound tree.

### 2.3 No new migrator (yet)

The v1 draft proposed `calor fix --elide-single-arg-calls`. Defer that until
emitter behavior is settled and a measured corpus sweep shows it is worth
the maintenance cost. The emitter change covers all freshly-emitted code.

## 3. Compatibility and disambiguation

### 3.1 Backward compatibility

Existing programs are unchanged. The closer-explicit forms remain accepted.
This RFC only **relaxes** the grammar.

### 3.2 Disambiguation (the central design question)

The grammar relaxation introduces three real ambiguities. Calor expressions
are Lisp-prefix (`(+ a b)`, `(- x y)`) not infix, so the v1 draft's
`§C{f} a + b` example does **not** apply — there is no infix operator in
Calor that would create the C#-style precedence problem. The actual
ambiguities are:

**A. Trailing postfix on the argument vs the call result**

```calr
§B{x} §C{Box.unwrap} y.Length
```

Two readings:

| Reading | Parse |
|---|---|
| A | `x = Box.unwrap(y.Length)`  (`.Length` attaches to `y`) |
| B | `x = Box.unwrap(y).Length`  (`.Length` attaches to the call result) |

Calor's existing `ParseTrailingMemberAccess` (Parser.cs:2756) attaches
trailing `.Member` to the immediately preceding expression. Adopt
**Reading A**: `.Length` attaches to `y`, the inline argument. For Reading
B the user writes `§C{Box.unwrap} y §/C.Length` (explicit closer).

**B. Multiple consecutive expression-start tokens on the same line**

```calr
§B{x} §C{Foo.bar} a b
```

Is `b` the second argument (which would require `§A` and `§/C` today) or
the start of a separate construct? The implementation rule: a closer-less
inline-arg call consumes **exactly one** primary expression. Any second
expression-start token after the first inline arg is a parse error
(`Calor0150 InlineArgUnexpected` — quick-fix wraps both in `§A … §A … §/C`).

**C. Nested closer-less calls**

```calr
§B{x} §C{Foo.bar} §C{Baz.qux} y
```

Reading A: `Foo.bar(Baz.qux(y))` — the inner call is the inline arg of
the outer; the inner consumes `y` as its inline arg; both end at newline.
This composes cleanly because the inner call is a primary expression.
Adopt Reading A.

**Diagnostic:**

| Code | Title | When |
|---|---|---|
| `Calor0150` | `InlineArgUnexpected` | Two expression-start tokens follow a closer-less call on the same logical line. Quick-fix inserts `§A`/`§/C`. **Severity: error.** |

**Alternatives considered:**

1. *Forbid closer elision entirely in nested or trailing-member contexts.*
   Cleaner spec but rejects 100 % of legitimate `§C{f} y.Length` uses.
   Rejected — too restrictive for the win.
2. *Greedy: consume every expression-start token as a positional argument
   until newline.* Matches the meaning users probably intended in case B
   but creates surprises with member access and nested calls. Rejected —
   `Calor0150` as error keeps the explicit `§A … §/C` form the canonical
   way to express multi-arg.
3. *Defer the RFC until expressions are infix.* Calor has no plans to add
   infix; rejecting on those grounds amounts to rejecting the RFC.

## 4. Measured impact (under benchmark tokenizer)

Per §1.4, dropping `§/C` saves 3 tokens and dropping `§A` saves 2 tokens.
Each transformation, applied to the corpus, would save:

| Transformation | Eligible calls | Tokens saved (per call) | Tokens saved (total) |
|---|---:|---:|---:|
| Drop `§/C` on zero-arg calls (currently `§C{x} §/C` → `§C{x}`) | 56 | 3 | **168** |
| Drop `§/C` on one-arg calls (`§C{x} §A y §/C` → `§C{x} §A y`) | 261 | 3 | **783** |
| Drop `§A` on one-arg calls (`§C{x} §A y` → `§C{x} y`) — already parser-legal for statement context | up to 261 | 2 | **up to 522** |
| **Subtotal (this RFC only — drop `§/C` only)** | 317 | 3 | **951** |
| **Subtotal (this RFC + emitter §A-drop, separate proposal)** | 317 + 261 | — | **up to 1 473** |

Total corpus tokens under the benchmark tokenizer: **162 961**.

| Scope | Saving | % of corpus |
|---|---:|---:|
| Drop `§/C` only (this RFC) | 951 | **0.58 %** |
| Drop `§/C` + emitter `§A`-drop | 1 473 | **0.90 %** |

The September benchmark's `TokenEconomics` is `1.122` (winner: calor,
significant, `pValue = 0.0001`), reported with a tight 95 % CI in
`website/public/data/benchmark-results.json`. A 0.58–0.90 % corpus-level
saving should push the ratio to approximately **1.13×** — a small but
measurable improvement.

Note on attribution: the histogram in §1.3 does not split statement vs
expression calls. The 261 one-arg calls include both contexts. The
emitter `§A`-drop column is "up to" because it requires a separate
emitter change to actually adopt the implicit-close form that
Parser.cs:1374-1382 already supports for statement context.

Beyond the corpus, the change pays at every AI-generated call site at
runtime — each elided `§/C` is 3 fewer LLM output tokens, each elided
`§A` is 2 fewer. That gain is not in the corpus measurement.

A benchmark gate is reasonable; the expected ~1 % improvement (see §8)
should be detectable by the 30-iteration statistical suite.

## 5. Risks

1. **Parser-precedence surprise** (see §3.2). Mitigation: Reading A + a
   hard error diagnostic `Calor0150` for ambiguous continuations. If
   reviewers reject Reading A, defer the RFC.
2. **Three-form proliferation** (explicit, inline-with-closer, closer-less).
   Mitigation: documentation; emitter default to the most concise legal
   form; cookbook entry.
3. **Migration of existing code that uses `§A (+ x y) §/C` (Lisp-prefix
   argument).** Mitigation: such arguments are already primary expressions
   (the entire `(...)` parenthesised form), so closer-less migration is
   semantics-preserving. The deferred migrator must still verify by
   re-parsing the rewritten form.
4. **Snapshot churn** in `tests/Calor.Conversion.Tests/TestData/`. Mitigation:
   gate the emitter behind `CalorEmitterOptions.UseImplicitCallCloser`
   (default `false` in v0.6.0, opt-in via flag for one release; flip in v0.6.1
   after consumers have updated snapshots).

## 6. Implementation plan

1. **Phase 0** — Parser tests pinning current behavior (statement implicit
   close works, expression implicit close fails today). No code changes.
2. **Phase 1** — Parser: in `ParseCallExpression` (line 7830), after the
   `ParseAttributes()` block, if the next token is a syntactic terminator
   (per §2) accept zero-arg + no `§/C`; else if `IsExpressionStart()` and
   `_position` is at primary precedence, accept one inline arg + no `§/C`.
   Emit `Calor0150` when a binary operator follows.
3. **Phase 2** — Emitter: gate elision behind
   `ConversionContext.UseImplicitCallCloser` (default `false` in v0.6.0,
   `true` in v0.6.1).
4. **Phase 3** — Update `docs/syntax-reference/calls.md` to document **both**
   the existing statement-context implicit close (currently undocumented) and
   the new expression-context closer elision.
5. **Phase 4** — Measure on the benchmark corpus. Expected `TokenEconomics`
   improvement is **~1.13×** (vs current 1.122×). Track both the benchmark
   delta and the fraction of calls round-tripped via the elided form (target:
   ≥ 90 % of eligible calls).

## 7. Open questions

- Should we extend the inline-arg form to allow a single `?` postfix
  (null-conditional) without parens? Probably yes — `?` is a postfix
  primary modifier, no ambiguity.
- Should we deprecate the explicit `§/C` for one-arg expression calls in
  a future release? **No.** Both forms remain canonical; the emitter
  picks the shorter when round-trip allows.

## 8. Decision criteria

Ship in v0.6.1 only if **all four** hold:

- Reviewers accept Reading A (§3.2).
- Parser tests for `Calor0150` give clear error messages on the
  multi-expression and trailing-postfix ambiguities (e.g. `§C{f} a b` and
  `§C{Box.unwrap} y.Length`).
- `RoundTrip.Harness` reports zero regressions across all snapshots when
  the emitter flag is enabled.
- Benchmark `TokenEconomics` shows a positive movement (lower 95 % CI
  bound above the current `1.122`). The v1 draft's `≥ 1.15` bar was based
  on fabricated counts and is reduced to "measurably positive".
