# RFC: Formalize and harden the existing `§B` type inference

**Status:** Draft (v2 — rewritten after architectural-fiction audit; see §0.1)
**Owner:** Calor language team
**Target release:** v0.6.x
**Companion RFCs:** `v0.6-call-closer-elision.md`

## 0. Front matter

### 0.1 Correction note (v2)

The v1 draft of this RFC ("Token Economics v1 — Type inference for `§B`
binding initializers") proposed type inference for `§B` as if it were a new
feature. A code audit showed inference is **already implemented** in
`Binder.cs:259-269`, the parser already accepts the colon-less form
(`Parser.cs:4350, 4390`), and the corpus already uses the inferred form for
**~85 % of bindings** (576 total, only 86 with a colon-form annotation).

What changed in v2:

- **Scope reframed** from "add inference" to "document and harden the
  existing inference":
  1. The behavior is undocumented in `docs/syntax-reference/`.
  2. The fallback for `§B{x}` with no type **and** no initializer silently
     defaults to `INT` — a latent bug.
  3. The inference rule is shallow and partial; specifying it explicitly
     makes future extensions safer.
- **All frequency counts replaced** with measured values and reproducible
  commands (see §2).
- **Token-economics estimate corrected** from the v1 "+8 % `TokenEconomics`"
  to a **measured corpus saving of up to 172 tokens out of 162 961 corpus
  tokens (~0.11 %)** under the September benchmark tokenizer (whitespace +
  punctuation; see §2 for the tokenizer reference). The v1 draft should
  not be used as design or measurement evidence.

### 0.2 Why this RFC still exists after the correction

The v1 motivation ("inference would be a win") is gone. The v2 motivation is:

1. **Documentation gap.** Inference works today but is undocumented; agents
   that read `docs/syntax-reference/binding.md` to learn Calor will not learn
   about it. The corpus is using a feature nobody is told about.
2. **Silent-INT bug.** `§B{x}` with no type and no initializer is currently
   bound as `INT` with no diagnostic. That is wrong by any reasonable
   reading and is a real correctness issue for AI-generated code.
3. **Locality guarantee.** Today's rule is "use `initializer.TypeName` if
   present". This RFC specifies that intentionally (per §3) so that any
   future extension goes through a design review rather than silently
   widening what compiles.

## 1. Existing behavior (verified against the parser + binder source)

### 1.1 Parser accepts colon-less `§B{name}` and `§B{~name}`

`src/Calor.Compiler/Parsing/Parser.cs:4344-4391`:

```csharp
private BindStatementNode ParseBindStatement()
{
    var startToken = Expect(TokenKind.Bind);
    var attrs = ParseAttributes();
    var (name, isMutable, typeName) = AttributeHelper.InterpretBindAttributes(attrs);
    ...
    return new BindStatementNode(span, name, typeName, isMutable, initializer, attrs);
}
```

When the attribute block is `{x}`, `AttributeHelper.InterpretBindAttributes`
returns `typeName == null`. The node is constructed without complaint.

### 1.2 Binder infers from the initializer

`src/Calor.Compiler/Binding/Binder.cs:257-272`:

```csharp
private BoundBindStatement BindBindStatement(BindStatementNode bind)
{
    var typeName = bind.TypeName ?? "INT"; // Default to INT if not specified
    BoundExpression? initializer = null;

    if (bind.Initializer != null)
    {
        initializer = BindExpression(bind.Initializer);
        if (bind.TypeName == null)
        {
            typeName = initializer.TypeName;   // ← INFERENCE TODAY
        }
    }

    var variable = new VariableSymbol(bind.Name, typeName, bind.IsMutable);
    ...
}
```

Two paths exist:

| Source | `bind.TypeName` | Initializer | Bound type |
|---|---|---|---|
| `§B{x:i32}` | `"i32"` | — | `i32` |
| `§B{x:i32} INT:0` | `"i32"` | `0:INT` | `i32` |
| `§B{x} INT:0` | `null` | `BoundIntLiteral` (`TypeName == "INT"`) | **`INT`** *(inferred)* |
| `§B{x} STR:"hi"` | `null` | `BoundStringLiteral` (`TypeName == "STRING"`) | **`STRING`** *(inferred)* |
| **`§B{x}` (no init, no type)** | **`null`** | **none** | **`INT` (silent default — bug)** |

Inferred types come from `BoundExpression.TypeName`, which uses the
binder's **uppercase internal names** (`INT`, `STRING`, `BOOL`, `FLOAT`,
etc. — see `src/Calor.Compiler/Binding/BoundNodes.cs:313`). This RFC
documents that behavior as-is rather than proposing a rename; any
public-facing rename (e.g. surfacing `i32` / `str` / `bool` / `f64` to
hover or docs) would be a separate change.

The fourth-row silent-`INT` default is the existing latent bug.

## 2. Corpus measurement (reproducible)

```powershell
$files = Get-ChildItem -Path samples,tests -Recurse -Filter *.calr -File
$binds       = ($files | ForEach-Object { (Get-Content $_ -Raw) } |
                ForEach-Object { ([regex]::Matches($_, '§B\{')).Count } | Measure-Object -Sum).Sum
$bindsColon  = ($files | ForEach-Object { (Get-Content $_ -Raw) } |
                ForEach-Object { ([regex]::Matches($_, '§B\{[^}]*:[^}]*\}')).Count } | Measure-Object -Sum).Sum
```

Run against the repo at the commit this RFC was prepared against
(`docs/tokenecon-v1-rfcs` branch, rebased on `10583118`):

| Quantity | Value |
|---|---:|
| `.calr` files in `samples/` + `tests/` | 1 552 |
| `§B{` openers | **576** |
| `§B{…:…}` colon-form | **86** |
| Inferred-form share | **490 / 576 = 85.1 %** |
| Explicit-annotation share | **14.9 %** |

The v1 claim "≈98 % of bindings carry annotations today" was the **inverse**
of reality — the actual figure is ~15 %. The four-row "histogram of
initializer shapes" in v1 §4 (61/22/9/5/3) was also fabricated; the binder
does not classify initializers that way and the corpus was never sampled.

A representative sample of inferred-form bindings drawn from the corpus
(`rg --no-line-number '§B\{[^:}~]+\}' samples/ tests/ | head`) shows:

```calr
§B{score} 85
§B{result} (- 0 n)
§B{~result} 1
§B{r} (% n 10)
§B{dx} (- x2 x1)
```

All five are inferred today by the binder rule in §1.2.

### 2.1 Benchmark tokenizer

`TokenEconomics` uses
`src/Calor.Compiler/Evaluation/Metrics/TokenEconomicsCalculator.cs`
`TokenizeSource`, which splits on whitespace **and** on individual
punctuation/symbol characters. Under that tokenizer:

| String | Tokens |
|---|---:|
| `§B{x:i32}` | 7 (`§`, `B`, `{`, `x`, `:`, `i32`, `}`) |
| `§B{x}` | 5 (`§`, `B`, `{`, `x`, `}`) |

So dropping `:i32` from a binding header saves **2 tokens** under the
benchmark tokenizer. The benchmark corpus tokenized under the same rules
contains **162 961** tokens across the 1 552 `.calr` files. These two
numbers anchor every estimate in §4.

## 3. Proposal

### 3.1 Document the existing rule

Add `docs/syntax-reference/binding.md` (new) with the rule table from §1.2,
plus the literal-type table (using the **uppercase internal names** the
binder actually produces — see `BoundNodes.cs`):

| Literal node | `TypeName` |
|---|---|
| `BoundIntLiteral` (`INT:42`) | `INT` |
| `BoundStringLiteral` (`STR:"x"`) | `STRING` |
| `BoundBoolLiteral` (`BOOL:true`) | `BOOL` |
| `BoundFloatLiteral` (`FLOAT:3.14`) | `FLOAT` |
| `BoundCallExpression` (`§C{Foo.bar} … §/C`) | declared return type of `Foo.bar` |
| `BoundReference` (identifier) | declared type of the referenced symbol |
| `BoundBinaryOperation` (`(+ a b)`, etc.) | result type of the binary op (see `BindBinaryOperation`) |
| `BoundNoneLiteral` (`none`) | declared via `NoneExpressionNode.TypeName` if present; otherwise inference fails (§3.2) |

Promote the document under `docs/syntax-reference/` so it is reachable from
the syntax-reference index. The doc names the internal types as-is; if the
project later decides to surface lowercase aliases (`i32`, `str`, etc.) to
users, that is a separate, larger change.

### 3.2 Replace the silent `INT` default with a diagnostic

`Binder.cs:259` currently reads:

```csharp
var typeName = bind.TypeName ?? "INT"; // Default to INT if not specified
```

Replace with:

```csharp
var typeName = bind.TypeName;
if (typeName == null && bind.Initializer == null)
{
    _diagnostics.Report(Calor0250_BindRequiresTypeOrInitializer, bind.Span, bind.Name);
    typeName = "INT"; // continue with an arbitrary type so binding still resolves
}
```

The continue-with-INT behavior is preserved so downstream binding does not
cascade; the diagnostic is the user-visible change.

Diagnostic codes (allocated in the semantic range `Calor0200`–`Calor0299`,
verified clear at the time of writing via
`rg 'Calor025[0-3]' src/Calor.Compiler` → no matches):

| Code | Title | Severity | When |
|---|---|---|---|
| `Calor0250` | `BindRequiresTypeOrInitializer` | **error** | `§B{x}` with no `:type` and no initializer. |
| `Calor0251` | `BindCannotInferNullLiteral` | **error** | `§B{x} null` / `§B{x} none` — no type to infer from. Quick-fix suggests `§B{x:Option<T>} none`. |
| `Calor0252` | `BindCannotInferGenericReturn` | **error** | `§B{x} §C{Vec.empty}` where the call returns a generic with unresolved type parameter. Quick-fix suggests `§B{x:Vec<T>} §C{Vec<T>.empty}`. |
| `Calor0253` | `BindAmbiguousNumeric` | **error** | `§B{x} (+ INT:0 FLOAT:0.0)` where widening could pick more than one type. Quick-fix suggests an explicit `:i32` or `:f64`. Uniform severity with `0250-0252`; the v2 draft initially marked this as a warning, but consistency outweighs the marginal cost of an explicit annotation in genuinely ambiguous cases. |

Only `Calor0250` is **required** for v0.6 (it is the bug-fix). The other
three are **optional extensions** that should ship together with their
quick-fixes in the LSP, or be deferred to v0.7.

### 3.3 Out of scope for v2

- No new parser grammar (the colon-less form already parses).
- No new emitter behavior. `CalorEmitter` already emits the colon-less form
  when the source had it; round-trip is unchanged.
- No `calor fix --infer-bind-types` migrator. The corpus measurement shows
  most explicit annotations are on bindings without initializers or with
  generic calls (`§B{xs:Vec<i32>} §C{Vec.empty}` style) where inference
  would deliberately fail per §3.2. Migration is not worth the maintenance
  cost until §3.2 quick-fixes prove out.

## 4. Token-economics estimate (corrected, benchmark tokenizer)

Per §2.1, dropping `:i32` from a binding header saves 2 tokens under the
September benchmark tokenizer. The maximum possible saving if every
existing colon-form annotation were inferred away:

| Source | Annotations | Tokens saved (per binding) | Total |
|---|---:|---:|---:|
| 86 colon-form bindings in corpus | 86 | 2 | **172** |

Realistic saving (after excluding annotations the inference rule cannot
recover — initializer-less `§B`s, generic calls, ambiguous numerics):
probably **~100 tokens**, against a corpus of **162 961 tokens**. That is
**~0.06 %** — at or below the detection edge of the 30-iteration
benchmark suite (September `TokenEconomics = 1.122` with a tight CI as
published in `website/public/data/benchmark-results.json`).

This RFC's value is **not** primarily token-economics. It is:

1. **Correctness**: fix the silent-`INT` bug (`Calor0250`).
2. **Discoverability**: document a feature already in use by 85 % of the
   corpus.
3. **Locality**: pin the inference rule so future extensions are explicit.

Do not gate v0.6 release on a `TokenEconomics` improvement from this RFC.

## 5. Risks

1. **Existing programs that depend on the silent `INT` default break.**
   Mitigation: the silent-default path is currently unreachable from any
   valid Calor program in the corpus (`§B{x}` always has either a type or
   an initializer). Verified by:

   ```powershell
   rg '§B\{[a-zA-Z_~][a-zA-Z0-9_]*\}\s*$' samples/ tests/   # match §B{x} at end of line
   ```

   → zero matches. Risk-of-break: zero in the corpus, low in the wild.

2. **`Calor0251`–`Calor0253` may surface in code that previously compiled.**
   Mitigation: ship `Calor0250` (the bug-fix) in v0.6.0; ship the rest in
   v0.6.1 behind a `--strict-bind-inference` flag for one release.

3. **Documentation drift.** Once the inference rule is documented, every
   future change to `BindBindStatement` must update the document.
   Mitigation: a unit test that asserts the documented examples bind to
   the documented types (`tests/Calor.Semantics.Tests/BindInferenceDocsTests.cs`).

## 6. Implementation plan

1. **Phase 0** — Write `tests/Calor.Semantics.Tests/BindInferenceDocsTests.cs`
   that pins every row of the §1.2 table to the current binder behavior.
   These are characterization tests; no production code changes yet.
2. **Phase 1** — Write `docs/syntax-reference/binding.md` and reference it
   from the syntax-reference index.
3. **Phase 2** — Add `Calor0250 BindRequiresTypeOrInitializer` and the
   replacement of the silent-`INT` default. One file change in `Binder.cs`,
   one new diagnostic registration, one unit test.
4. **Phase 3** — *(Optional, v0.6.1)* Add `Calor0251`–`Calor0253` with LSP
   quick-fixes behind `--strict-bind-inference`. Measure on benchmark
   corpus; promote to default only if no regressions.

## 7. Open questions

- Should `Calor0250` be promoted from warning to error in v0.7? Probably
  yes — the silent-`INT` default has no defensible use case.
- Should we extend inference to peer at chained calls
  (`§B{x} §C{a.b} §A foo §/C.length`)? Out of scope; the existing rule
  handles only the outermost initializer's type.
- Should `§B{x} §IF (cond) → §R y` infer `y`'s type? Currently no — the
  binder skips the `§IF` initializer. Defer to a separate RFC.

## 8. Decision criteria

Ship `Calor0250` in v0.6.0 if:

- `tests/Calor.Semantics.Tests/BindInferenceDocsTests.cs` is green.
- The full test suite is green after replacing the silent default.

Defer §3.2's `Calor0251`–`Calor0253` to v0.6.1 unless their LSP quick-fixes
are also ready.

The corpus-scale token-economics gate from the v1 draft
(`TokenEconomics ≥ 1.20`) is removed — that bar was based on fabricated
counts and is not achievable from this RFC.

## 9. Combined effect with companion RFC (honest version)

| RFC | Corpus tokens saved (benchmark tokenizer) | Notes |
|---|---:|---|
| v6 compact IDs (PR #641, shipped) | 0 | Corpus uses test IDs (`m001`), not ULIDs. Verified zero movement in the September benchmark. Real gain only on repos with production ULIDs to migrate. |
| Call closer elision (companion) | 951 (~0.58 %), up to 1 473 (~0.90 %) if emitter also drops `§A` | Plausibly detectable in the 30-iteration benchmark; value is mostly in design hygiene + AI runtime cost. |
| Bind inference formalization (this RFC) | ≤ 172 (~0.11 %), realistic ~100 (~0.06 %) | Near or below benchmark detection edge; value is in bug-fix + documentation. |

Combined corpus impact: ≤ ~1 645 tokens / 162 961 total = **~1.0 %**. With
the September `TokenEconomics = 1.122`, a fully-realised combined effect
would land near **1.13×** — a modest but measurable win.

**A more honest framing for v0.6:** correctness + documentation +
ergonomics + a small (~1 %) token-economics improvement. A *large*
token-economics win still requires a structural change (e.g. compact-ID
adoption on a ULID-bearing external corpus, or a wholly new compact
syntax for a different high-frequency construct), not the three
optimisations in this v1 program.
